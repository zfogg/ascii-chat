ascii-chat(5) File Formats ascii-chat(5)

NAME

ascii-chat - file format specifications for ascii-chat configuration and authentication

DESCRIPTION

ascii-chat uses several configuration and data files. This manual describes the format and location of each file type.

FILES

Configuration Files

~/.config/ascii-chat/config.toml
Main configuration file in TOML format. See CONFIG.TOML FORMAT below for details. Default location on Linux/macOS is $XDG_CONFIG_HOME/ascii-chat/config.toml (usually ~/.config/ascii-chat/config.toml). On Windows: %APPDATA%\ascii-chat\config.toml
~/.config/ascii-chat/colors.toml
Color scheme configuration in TOML format. See COLORS.TOML FORMAT below for details. Built-in color schemes are available and can be selected with the --color-scheme option.

Authentication Files

~/.ascii-chat/known_hosts
SSH-style host verification file. See KNOWN_HOSTS FORMAT below for details. Stores server public keys for TOFU (Trust On First Use) verification.
/path/to/authorized_keys
Client public key whitelist for server-side authentication. See AUTHORIZED_KEYS FORMAT below for details. Can be specified with the --client-keys option on the server.

SSH Key Files

~/.ssh/id_ed25519
SSH Ed25519 private key. See SSH ED25519 PRIVATE KEY FORMAT below for details. Used with --key option for server identity.
~/.ssh/id_ed25519.pub
SSH Ed25519 public key. See SSH ED25519 PUBLIC KEY FORMAT below for details. Used with --server-key option for client-side server verification.

GPG Key Files

~/.gnupg/pubring.gpg
GPG public keyring. See GPG KEY FORMAT below for details. Used for GPG-based key authentication.

CONFIG.TOML FORMAT

The main configuration file uses TOML (Tom's Obvious, Minimal Language) syntax. Configuration files are searched in order and merged together (cascade semantics). This means multiple config files can be used, with user config taking precedence over system defaults.

Search Order

On Unix/Linux and macOS, the following locations are searched in order:

1.
$XDG_CONFIG_HOME/ascii-chat/config.toml (usually ~/.config/ascii-chat/config.toml) - user config directory
2.
/opt/homebrew/etc/ascii-chat/config.toml macOS Homebrew installation directory
3.
/usr/local/etc/ascii-chat/config.toml Unix local installation directory
4.
/etc/ascii-chat/config.toml system-wide defaults On Windows, the following locations are searched in order:
1.
%APPDATA%\ascii-chat\config.toml user configuration directory
2.
%PROGRAMDATA%\ascii-chat\config.toml system-wide defaults

Structure

The file is organized into sections:

[logging]
Log level and output settings
[general]
General application settings (sleep prevention, etc.)
[configuration]
Configuration file paths and creation settings
[terminal]
Terminal display settings (width, height, color scheme)
[webcam]
Webcam input settings (device index, flip, test pattern)
[audio]
Audio capture and playback settings (device indices, codecs)
[display]
Display rendering settings (aspect ratio, framerate, snapshots)
[network]
Network configuration (port, compression, WebRTC, discovery)
[security]
Encryption and authentication settings (keys, passwords, identity verification)
[media]
Media streaming settings (file/URL input, looping, cookies)
[database]
Discovery service database and persistence settings

Common Options

Options in config.toml are case-sensitive and use underscores for word separation (e.g., compression_level not compressionLevel ). Boolean values are true or false (lowercase). Lists are TOML arrays: [item1, item2, item3]

Example Section

[network]
compression_level = 3                 # zstd level (1-9)
discovery = false                     # Enable discovery session registration
[terminal]
color_scheme = "pastel"               # Color scheme (pastel, nord, dracula, etc.)
[display]
fps = 60                              # Target framerate (1-144)
[security]
encrypt = true                        # Enable end-to-end encryption

COLORS.TOML FORMAT

Custom color scheme configuration file using TOML syntax. Specifies RGB colors for different message log levels in both dark and light mode variants. The colors.toml file uses override semantics, meaning only the first matching color scheme file in the search path is loaded (user schemes take precedence over system-wide defaults). This file is optional; ascii-chat includes several built-in color schemes that can be selected with the --color-scheme command-line option. Custom color schemes are stored alongside config.toml using the same search path.

Structure

[scheme]
Metadata about the color scheme
[colors.dark]
Colors for dark background terminals
[colors.light]
Colors for light background terminals (optional)

Color Levels

Eight color slots are supported:

dev
Development/trace level messages (lowest severity)
debug
Debug level messages
warn
Warning level messages
info
Information level messages
error
Error level messages
fatal
Fatal error messages (highest severity)
grey
Auxiliary and metadata text
reset
Color reset/default (used between messages)

Color Format

Colors are specified as hexadecimal RGB values in format #RRGGBB where RR, GG, BB are two-digit hexadecimal values (00-FF) for red, green, and blue respectively.

Example Structure

[scheme]
name = "custom"
description = "My custom color scheme"
[colors.dark]
dev    = "#6b7fff"
debug  = "#65ace1"
warn   = "#f0cc91"
info   = "#90e070"
error  = "#e85d6f"
fatal  = "#c8a0d8"
grey   = "#808080"
reset  = "#ffffff"
[colors.light]
dev    = "#4b5fdf"
debug  = "#3284b4"
warn   = "#b48c00"
info   = "#238322"
error  = "#b21e2c"
fatal  = "#6e4c65"
grey   = "#4c4c4c"
reset  = "#000000"

Built-in Schemes

The following color schemes are built-in and available via the --color-scheme option without needing a colors.toml file:

pastel
Soft pastel colors (default) - auto-adapts to dark/light backgrounds
nord
Arctic, muted Nord theme - auto-adapts to dark/light backgrounds
solarized
Solarized theme - precision colors - auto-adapts to dark/light backgrounds
dracula
Dracula dark theme - vampiric colors (dark only)
gruvbox
Gruvbox theme - retro warm colors - auto-adapts to dark/light backgrounds
monokai
Popular Monokai dark theme (dark only)
base16-default
Base16 Default theme (Chris Kempson) - auto-adapts to dark/light backgrounds

KNOWN_HOSTS FORMAT

SSH-style host verification file for storing server public keys used in TOFU (Trust On First Use) identity verification. The known_hosts file is searched across multiple standard locations using append semantics, meaning entries from all matching files are combined together. The file uses the same search paths as config.toml (see CONFIG.TOML FORMAT above for details). On Unix/macOS, this includes ~/.config/ascii-chat/known_hosts, /opt/homebrew/etc/ascii-chat/known_hosts, /usr/local/etc/ascii-chat/known_hosts, and /etc/ascii-chat/known_hosts. On Windows, %APPDATA%\ascii-chat\known_hosts and %PROGRAMDATA%\ascii-chat\known_hosts are searched. The first file (user config directory) is created by ascii-chat automatically when servers are visited.

Line Format

Each line represents a single identity entry with the format:

<address> <key-type> <public-key> [comment]

Where:

<address>
Server address with port in format host:port For IPv6 addresses, use bracket notation: [2001:db8::1]:8080
<key-type>
Always x25519 for elliptic curve Diffie-Hellman keys
<public-key>
64-character hexadecimal representation of the 32-byte public key
[comment]
Optional comment (space-separated from key). Typically contains the server name or connection context.

Special Entry

A special entry indicates no identity was obtained:

<address> no-identity [comment]

This records that a connection was attempted but no server identity could be obtained (e.g., ephemeral DH without key authentication).

Example File

192.0.2.1:27224 x25519 a1b2c3d4e5f6g7h8 ascii-chat-server
example.com:27224 x25519 f8e7d6c5b4a39291 example-prod-server
[2001:db8::1]:27224 x25519 1a2b3c4d5e6f7g8h ipv6-server
10.0.0.5:8080 no-identity temp-test-server

AUTHORIZED_KEYS FORMAT

Server-side client public key whitelist, one SSH Ed25519 public key per line. Can be specified with --client-keys /path/to/file on the server.

Line Format

Standard OpenSSH public key format, one key per line:

ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIxxx... [comment]

Where:

ssh-ed25519
Key type identifier (always ssh-ed25519 for Ed25519 keys)
AAAAC3Nz...
Base64-encoded public key data
[comment]
Optional comment (e.g., user@hostname or description)

Usage

When a server is started with --client-keys /path/to/file , only clients whose public keys appear in the file are allowed to connect.

Example File

ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIzTVYwABZIE2mOEZzzV3 alice@workstation
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIa0HxPmBfLjp3WkNXx bob@laptop
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIsK4L9NjTqZWF2bxVp charlie@device

SSH ED25519 PUBLIC KEY FORMAT

Standard OpenSSH public key format, typically stored in ~/.ssh/id_ed25519.pub

Format

ssh-ed25519 <base64-encoded-key> [<comment>]

Where:

ssh-ed25519
Key type identifier for Ed25519 elliptic curve
<base64-encoded-key>
Base64 encoding of the 32-byte public key
[<comment>]
Optional comment, typically user@hostname

Usage

--server-key
Verify server identity by providing the server's public key
--client-keys
Whitelist of allowed client public keys for server-side authentication
github:<username>
Fetch user's public keys directly from GitHub (keys are public)

Example

ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIzTVYwABZIE2mOEZzzV3 user@example.com

SSH ED25519 PRIVATE KEY FORMAT

OpenSSH private key format with Ed25519 algorithm, typically stored in ~/.ssh/id_ed25519

File Structure

-----BEGIN OPENSSH PRIVATE KEY-----
<base64-encoded-data>
-----END OPENSSH PRIVATE KEY-----

Encryption

Private keys can be optionally encrypted with a passphrase. ascii-chat will prompt for the passphrase if the key is encrypted. Use the passphrase parameter or environment variable ASCII_CHAT_KEY_PASSWORD (not recommended - use ssh-agent instead).

File Permissions

Private keys should have restrictive permissions (0600):

chmod 600 ~/.ssh/id_ed25519

ascii-chat requires SSH key files to have restrictive permissions (0600 or 0400). If permissions are more permissive, ascii-chat will refuse to use the key and return an error. Run chmod 600 ~/.ssh/id_ed25519 to fix the permissions.

Usage

--key
Server identity key (specifies private key path)

GPG KEY FORMAT

OpenPGP armored public key format, typically from the GPG public keyring.

Format

-----BEGIN PGP PUBLIC KEY BLOCK-----
Version: GnuPG v2.X
<base64-encoded-key-data>
-----END PGP PUBLIC KEY BLOCK-----

Supported Keys

Only Ed25519 keys are supported. Legacy RSA and DSA keys will be rejected. Key IDs can be:

8 characters
Short key fingerprint
16 characters
Long key fingerprint
40 characters
Full key fingerprint

Requirements

gpg
Requires the gpg command-line tool to be installed and available in PATH
gpg-agent
For encrypted keys, requires GPG Agent for passphrase management (recommended over prompting)

Usage

--server-key
Specify GPG key with format: gpg:<key-id> or gpg:<fingerprint>

Example

--server-key gpg:ABCD1234
--server-key gpg:ABCD1234567890EF
